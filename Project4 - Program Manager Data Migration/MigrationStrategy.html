Strategic Imperative: The "migration bubble" — the period where Mimecast pays for both its private DC and AWS infrastructure simultaneously — is the single largest financial risk in this programme. All wave execution decisions are optimized to compress this window to under 8 weeks.

OPTIMIZATION PILLARS
  • Graviton3 instance adoption
  • Aurora Serverless v2 for variable DB workloads
  • EBS io2 → gp3 migration (−20% cost)
COMMERCIAL LEVERS

  • Compute Savings Plans (1-year, no-upfront)
  • S3 Intelligent-Tiering + Lifecycle Policies
GOVERNANCE CONTROLS
  • 15% daily spend spike auto-alert
  • Mandatory resource tagging policy (SCPs)
  • Weekly COGS review cadence

§ 02 · THE WAVE MODEL
12-Week Execution Plan — April 1 to June 23, 2026
Click Waves to Expand
W0 FOUNDATION
Wave 0 — Foundation & FinOps Scaffolding Weeks 1–2 · Apr 1–14 Infra Zero ▶

Strategic Focus: Establish FinOps guardrails before a single workload touches AWS. No engineering team ships infrastructure without tagging policies, budget alerts, and MAP credit enrollment in place. Cost attribution is non-negotiable from day one.

Week 1 · Apr 1–7
🏗 FinOps Infrastructure Setup
DB Expert Levers
  • Inventory all existing DB workloads: Oracle, MSSQL, PostgreSQL instance families, storage volumes, IOPS profiles
  • Capture CloudWatch Disk I/O baselines for all DBs to size gp3 provisioned IOPS correctly
  • Identify Aurora Serverless v2 candidates (variable-load, dev/test DBs)
  • Flag io2 volumes: projected savings migrating to gp3 at same IOPS (−20% cost)
AWS Commercial Levers
  • Enroll in AWS MAP — secure migration credits. Tag all new resources with map-migrated: yes immediately
  • Enable AWS Cost Explorer + Cost Anomaly Detection (15% daily threshold)
  • Configure AWS Budgets — per-team budget alerts at 80% and 100% thresholds
  • Activate Consolidated Billing under AWS Organization with SCPs for tagging enforcement
Week 2 · Apr 8–14
🏷 Tagging Policy & Baseline Baselining
DB Expert Levers
  • Deploy AWS DMS replication instances in target VPCs — use r7g (Graviton3) DMS instances for ~20% compute saving
  • Enable RDS Performance Insights on all target RDS instances from creation
  • Size DMS replication instance storage: use gp3 not gp2 for all DMS volumes
AWS Commercial Levers
  • Define mandatory tagging taxonomy: team, env, product, cost-center, migration-wave
  • Enforce via AWS Config + SCP — deny resource creation without required tags
  • Establish S3 Intelligent-Tiering baseline policy on all migration staging buckets
  • Capture DC total cost baseline (colocation + power + bandwidth) for bubble tracking
W1 ASSESS
Wave 1 — Assessment, Network & Non-Prod Migration Weeks 3–4 · Apr 15–28 Non-Prod First ▶

Strategic Focus: Migrate non-production workloads first to validate tooling, burn in monitoring, and generate real AWS cost data against which to right-size production. Network egress costs are profiled and Direct Connect pricing locked in before production cutover.

Week 3 · Apr 15–21
🔬 Right-Sizing & Non-Prod DB Migration
DB Expert Levers
  • Migrate dev/test databases using DMS full-load — validate schema conversion via AWS SCT
  • Run non-prod on Aurora Serverless v2 — set ACU min=0.5, max=8; validate autoscaling against actual dev traffic
  • Compare db.r7g (Graviton3) vs db.r6i on non-prod for benchmark lift before committing to prod
  • Enable RDS Storage Autoscaling for all non-prod instances to avoid over-provisioning
AWS Commercial Levers
  • Non-prod runs on Spot Instances wherever possible (dev environments, batch jobs, CI/CD runners)
  • Use AWS Compute Optimizer recommendations on first week of non-prod data
  • Tag all non-prod resources with env: non-prod; allocate separate budget alert
  • Evaluate AWS Reserved Instance Marketplace for 1-year standard RI purchases on steady-state services
Week 4 · Apr 22–28
🌐 Network Cost Profiling
DB Expert Levers
  • Measure DMS migration throughput — profile data transfer bytes and ingress costs (ingress to AWS is free; validate egress patterns)
  • Optimize DMS LOB column handling — Limited LOB mode vs Full LOB mode dramatically impacts replication cost
  • Profile read replica traffic — identify candidates for RDS Proxy to reduce connection overhead and costs
AWS Commercial Levers
  • Lock in AWS Direct Connect pricing for production bulk data transfer — dramatically cheaper than internet egress at scale
  • Configure VPC Flow Logs + cost attribution for cross-AZ data transfer monitoring
  • Evaluate S3 Transfer Acceleration vs Direct Connect for bulk DB dump transfer — benchmark cost/time
  • First weekly COGS review — compare actuals vs MAP-adjusted baseline
W2 DB MIG
Wave 2 — Production Database Migration Weeks 5–7 · Apr 29 – May 19 Critical Path ▶

Strategic Focus: Execute production database migrations — the highest-risk, highest-impact cost events. DB teams lead on Graviton3 adoption, Aurora I/O optimization, and EBS modernization. The "bubble" cost peak occurs here — compress it with aggressive DC rack exit planning running in parallel.

Week 5 · Apr 29 – May 5
🗄 Tier-1 DB Migration Begin
DB Expert Levers
  • Initiate DMS CDC (Change Data Capture) replication for Tier-1 databases — target lag ≤ 60 seconds before cutover
  • Provision production RDS instances on db.r7g.2xlarge (Graviton3) — validate against benchmark results from Wave 1
  • Migrate all EBS volumes on DB hosts from io2 → gp3 — match IOPS and throughput; lock in 20% cost reduction
  • Deploy Aurora I/O-Optimized configuration for high-throughput Aurora clusters — eliminates per-million I/O charge above baseline
AWS Commercial Levers
  • Purchase 1-year Compute Savings Plans (no-upfront) for steady-state RDS and EC2 instances — 30–40% vs On-Demand
  • Validate all DB resources tagged map-migrated: yes for MAP credit capture
  • Monitor cost anomaly threshold — production DBs will spike on initial provisioning; pre-register as scheduled exceptions
Week 6 · May 6–12
🔄 CDC Stabilization & Read Replica Optimization
DB Expert Levers
  • Validate CDC replication lag ≤ 60s consistently; tune DMS task settings: ParallelLoadBufferSize, CommitRate
  • Right-size Aurora read replicas using Performance Insights — kill over-provisioned replicas burning cost unnecessarily
  • Enable RDS Proxy on high-connection-count services — reduces connection overhead, enables costless connection pooling
  • Implement DB parameter group tuning: reduce innodb_buffer_pool_size on undersized instances, tune work_mem on PostgreSQL
AWS Commercial Levers
  • Review AWS Trusted Advisor "Cost Optimization" checks — act on idle DB instances, unattached EBS volumes, old snapshots
  • Implement RDS snapshot lifecycle policies — retain only 7 days of automated snapshots during migration; prune manual snapshots
  • Confirm Multi-AZ cost is justified — convert non-critical DBs to Single-AZ during migration window, restore post-cutover
Week 7 · May 13–19
🗃 Storage Tiering & Archival Migration
DB Expert Levers
  • Migrate cold / archival database tables to S3 + Athena pattern — remove from RDS tier entirely (eliminate DB storage cost for static data)
  • Implement S3 Intelligent-Tiering lifecycle policy: objects >30 days → IA tier; >90 days → Archive Instant Access
  • Set S3 Glacier Instant Retrieval for compliance data archives — 68% cheaper than S3 Standard
  • Audit and delete orphaned DB snapshots across all accounts — snapshots compound cost silently
AWS Commercial Levers
  • First MAP credit reconciliation — verify credits applied to EC2, RDS, DMS line items in Cost Explorer
  • Evaluate converting high-memory EC2 app tier to Graviton3 (M7g/R7g) — 20% better price/performance
  • Review cross-AZ data transfer costs in VPC Flow Logs — restructure service placement to minimize AZ-crossing where possible
W3 OPTIMIZE
Wave 3 — Application Migration & Cost Optimization Sprint Weeks 8–9 · May 20 – Jun 2 Pre Go-Live ▶

Strategic Focus: Application-tier migration and aggressive right-sizing before the May 31 Go-Live milestone. Every over-provisioned resource that survives cutover becomes permanent COGS drag. AWS Compute Optimizer recommendations are treated as actionable directives, not advisory noise.

Week 8 · May 20–26
⚡ Graviton3 Adoption & Right-Sizing
DB Expert Levers
  • Execute final cutover for all remaining production DBs — DMS CDC to zero-lag, then promote RDS replica to primary
  • Validate db.r7g Graviton3 production performance under full production load — rollback plan if p99 degrades >5%
  • Enable Aurora Serverless v2 autoscaling on eligible clusters — set aggressive ACU caps to prevent runaway scaling
AWS Commercial Levers
  • Act on all Compute Optimizer rightsizing recommendations with ≥10% cost saving — mandatory team sign-off if declining
  • Convert non-prod EC2 to Spot with Auto Scaling Groups using On-Demand base capacity = 1
  • Activate EC2 Instance Scheduler for non-prod — shut down dev environments 8 PM – 7 AM (saves ~60% of non-prod EC2 cost)
  • Purchase additional Savings Plans now that production shape is known — target 70% On-Demand coverage via Savings Plans
Week 9 · May 27 – Jun 2
🏁 Go-Live Prep & Bubble Compression
DB Expert Levers
  • Disable all DMS replication tasks post-cutover — replication instances are significant cost if left running idle
  • Terminate DMS replication instances within 24 hours of final task completion
  • Begin decommissioning DC-side DB servers in coordination with colocation provider SLA
  • Archive DC-side database backup tapes to S3 Glacier Deep Archive — lowest-cost long-term retention ($0.00099/GB/month)
AWS Commercial Levers
  • May 31 Go-Live — Begin DC decommissioning clock. Every day of delay = continued dual-run cost exposure
  • File rack exit notices with colocation provider — trigger contractual decommissioning process
  • Verify MAP credit burn rate in Cost Explorer — ensure credits are applying before commitment periods expire
W4 GO-LIVE
Wave 4 — Post Go-Live Stabilization & Continuous Optimization Weeks 10–11 · Jun 3–16 Stabilize ▶

Strategic Focus: Validate production stability on AWS, close the migration bubble by aggressively decommissioning DC assets, and embed continuous cost optimization into engineering team rituals — transitioning from "migration mode" to "steady-state FinOps mode."

Week 10 · Jun 3–9
📉 Bubble Close & DC Wind-Down
DB Expert Levers
  • Execute Aurora I/O cost audit — compare actual I/O billing vs I/O-Optimized break-even point per cluster
  • Run AWS Database Activity Streams audit to identify dormant schemas consuming storage unnecessarily
  • Validate backup retention policies — ensure RDS automated backups are set correctly (7 days production, 3 days non-prod)
AWS Commercial Levers
  • Formally close DC circuits and network connections that are no longer needed — Direct Connect ports not in use still incur charges
  • Review NAT Gateway costs in VPC — often the #1 hidden cost driver; evaluate PrivateLink or VPC Endpoints as replacement
  • Complete second MAP credit reconciliation and project remaining credit runway vs. actual spend velocity
  • Issue Divya's June 15 customer communication about the scheduled maintenance window
Week 11 · Jun 10–16
📣 Customer Comms & Pre-Maintenance Prep
DB Expert Levers
  • Pre-stage DB failover runbooks for June 30 maintenance window — validate multi-AZ automatic failover timing
  • Test Aurora fast failover — target <30 seconds RTO for all Tier-1 databases
  • Snapshot all production DBs 24 hours before maintenance window as rollback point
AWS Commercial Levers
  • Pre-purchase Savings Plans top-up based on 10-week production actuals — optimize coverage rate
  • Conduct Trusted Advisor Premium full scan — act on all "Cost Optimization" and "Performance" flags
  • Benchmark COGS: AWS run-rate vs pre-migration DC cost. Generate reduction % for executive report.
W5 DECOM
Wave 5 — Maintenance Window, Full Decommission & FinOps Steady State Week 12 · Jun 17–23 + Maint Window Jun 30 Final Phase ▶

Strategic Focus: Execute the June 30th Maintenance Window (10:30 PM–12:30 AM) as the final DC infrastructure cutover event. Achieve full DC rack exit. Transition to steady-state AWS FinOps operations with monthly cadence replacing the weekly migration war-room.

Week 12 · Jun 17–23
🔧 Maintenance Window Execution
DB Expert Levers
  • Jun 30, 10:30 PM — Freeze DC-side DB writes, execute final DMS sync, promote AWS RDS as authoritative primary
  • Verify all connection strings updated in application configs — no service should reference DC DB endpoints post-window
  • Run full integrity check on AWS databases post-cutover — row counts, constraint validation, index health
  • Power down DC DB servers — initiate hardware decommissioning with colocation provider
AWS Commercial Levers
  • Post-window: cancel all DC-related recurring costs — colocation contracts, cross-connect circuits, power allocations
  • Terminate all remaining DC-facing Direct Connect connections no longer needed post-decommission
  • Generate final Migration Bubble Cost Report — total dual-run period cost vs. projected; share with Eswar and exec team
  • Transition weekly COGS review to monthly FinOps review cadence — embed FinOps champions as permanent cost owners
Post Jun 30 · Steady State
♻️ FinOps Steady State Activation
DB Expert Levers
  • Implement Aurora auto-pause on all dev/test Aurora Serverless clusters (zero ACU billing when idle)
  • Schedule quarterly DB right-sizing reviews using 90-day CloudWatch utilization data
  • Implement S3 Lifecycle Rules to transition DB exports automatically — Standard → IA (30d) → Glacier (90d)
AWS Commercial Levers
  • Annual Savings Plans renewal review — evaluate converting flexible Compute SPs to EC2 Instance SPs (higher discount)
  • Implement AWS Cost Anomaly Detection ML models trained on 90 days of steady-state data — retire manual 15% rule for non-critical paths
  • Publish quarterly FinOps Scorecard — COGS per workload, Savings Plan coverage %, Compute Optimizer adoption rate
  • Target: DC fully dark by July 5, 2026 — zero colocation charges from this date
