§00 — Challenge

Engineering Excellence
DORA Elite in 90 Days

Mimecast's engineering velocity is plateauing. Slow CI/CD pipelines, fragmented tooling, and mounting tech debt are eroding developer experience and product quality. This initiative establishes a measurable, data-driven Engineering Excellence program — transforming the team from High/Medium DORA to Elite in 90 days.

Ashok Sakthivel · Sr. Director of Engineering
Mimecast · Bangalore Hub
April – June 2026
Target: DORA Elite Tier
⚠ Current State
  • Deployment frequency: 1–2×/week (DORA Medium tier)
  • Lead time for changes: 3–5 days from commit to prod
  • MTTR: 4+ hours — no standard incident runbook across teams
  • Change failure rate: 18% — flaky tests masking real failures
  • Tech debt backlog: 340 items, zero prioritisation framework
  • 3 different CI tools across 4 squads — no golden path
⚡ Root Causes
  • Pipeline Fragmentation: Each team built their own CI/CD — no shared standards
  • Flaky Test Debt: 12% of tests intermittently fail — blocking deploys daily
  • No Feature Flags: Big-bang releases instead of progressive rollout
  • Absent On-Call Discipline: No PagerDuty rotation; incidents pile on seniors
  • ADR Deficit: Architecture decisions undocumented — repeated mistakes recur
✅ Target State
  • Deployment frequency: Daily or multiple times/day (Elite)
  • Lead time: <1 hour from commit to production
  • MTTR: ≤30 minutes with automated runbook execution
  • Change failure rate: ≤5% — feature flags enable instant rollback
  • Golden Path CI/CD template adopted by all 4 squads
  • Tech debt sprint allocation: 20% capacity every sprint

Principle 1

One Pipeline to Rule Them All

Single GitHub Actions workflow template for all squads. Forks prohibited. Customisation via input params, not pipeline edits. Golden Path PRs reviewed by Platform Engineering.

Principle 2

Feature Flags Over Feature Branches

Every user-facing change shipped behind a LaunchDarkly flag. No big-bang releases. Progressive rollout: 1% → 10% → 50% → 100% with automated rollback on error rate spike.

Principle 3

Shift Security Left

Snyk SCA + CodeQL SAST run on every PR. Critical CVEs block merge. High CVEs create Jira P2 automatically. OWASP dependency check weekly on main branch.

Principle 4

Quarantine Flaky Tests

Tests that fail intermittently are automatically quarantined — they no longer block deploys but create Jira items with owners. Flaky rate measured weekly; team SLA: resolve in 2 sprints.

§07 — Anti-Pattern Registry

Engineering Excellence Failure Modes

Anti-PatternSeverityRoot CauseMitigation
🏚 Pipeline Sprawl
Each squad built its own CI/CD pipeline — 4 different tools, 4 sets of tribal knowledge, no shared maintenance
 High No Platform Engineering team; squads had full autonomy without a paved golden path
Golden Path CI/CD template mandatory. Old pipelines deprecated by Day 90. Waiver process exists but requires VP approval.
🎲 Flaky Test Tolerance
Teams mark flaky tests as "known issue" and skip them in CI — builds pass but quality signal is broken
 High No flaky test policy; skipping tests faster than fixing them; no ownership accountability
Quarantine, don't skip. Flaky tests auto-create Jira P2 with squad owner. 2-sprint SLA to fix. Flaky rate published weekly in engineering digest.
💣 Big-Bang Releases
Large feature branches merged infrequently — each release is high-risk, high-stress, and hard to roll back
 High No feature flag infrastructure; teams afraid of releasing incomplete features
LaunchDarkly mandatory for all user-facing changes. Trunk-based development with feature flags. Progressive rollout prevents blast radius.
🧠 Tribal Architecture Knowledge
Key architecture decisions exist only in senior engineers' heads — repeated mistakes, slow onboarding, fear of change
 Medium No ADR process; design review not documented; architecture review ad-hoc
ADR repository live — 10 critical decisions documented in Phase 3. All future significant decisions require an ADR before implementation PR is merged.
📉 Tech Debt Avoidance
Tech debt backlog treated as a "rainy day" list — it never gets allocated sprint capacity and grows unbounded
 Medium Product pressure always overrides debt allocation; no contractual sprint capacity agreement
20% sprint capacity contractually protected. Debt items scored via RICE. Failure to allocate = engineering lead escalates to SD of Eng. Published in squad scorecard.
🦸 On-Call Hero Culture
Same 2–3 senior engineers always paged — burnout risk, no knowledge transfer, single point of failure on incidents
 Medium No formal PagerDuty rotation; lack of runbook documentation; junior engineers excluded from on-call
PagerDuty rotation formalised — all L4+ engineers included. Automated runbooks for top-5 alert types. Buddy system: junior paired with senior for first 3 on-call weeks.
